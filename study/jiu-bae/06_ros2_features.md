# 6장. ROS1과 2의 차이점으로 알아보는 ROS2의 특징

## 6.1 ROS2 개발의 필요성

ROS1은 2007년경 개발이 시작된 이후, 로봇 연구·교육 분야에서 사실상 **표준 플랫폼**으로 자리 잡았습니다.  
하지만 최초 설계 당시의 전제 조건 때문에 다음과 같은 **제한 사항**을 가지고 있었습니다.

- 단일 로봇
- 워크스테이션급 컴퓨터
- 리눅스 환경
- 실시간 제어 미지원
- 안정된 네트워크 환경 요구
- 대학이나 연구소와 같은 아카데믹 연구 용도

ROS1은 2000년대 초반 개인 서비스 로봇인 PR2 개발을 위해 만들어진  
미들웨어 형태의 로봇 개발 프레임워크였습니다.  
이때 설정된 요구사항이 이후에도 오랫동안 유지되면서,  
오늘날 요구되는 로봇 개발 환경과는 점점 **괴리가 커지게 되었습니다.**

그 사이 ROS1은 학술 분야뿐 아니라,

- 제조 로봇
- 농업 로봇
- 드론
- 소셜 로봇

등 상용 로봇에도 폭넓게 사용되면서, 다양한 도메인에서 추가 요구사항이 등장했습니다.

**새로 등장한 요구사항 예시**

- 두 대 이상의 로봇
- 임베디드 시스템에서의 사용
- 실시간 제어
- 불안정한 네트워크 환경 대처
- 멀티 플랫폼
- 최신 기술 지원
- 상업용 제품 지원

이러한 요구사항을 ROS1에 그대로 반영하기에는 **구조적인 대규모 변경**이 필요했습니다.  
결국 기존 ROS1을 유지하면서, 차세대 기능을 담은 새로운 버전을 **ROS2**라는 이름으로  
분리하여 개발하게 되었습니다.

또한, 기존 자산을 보호하기 위해 **ROS1과 ROS2 간 통신이 가능한 브릿지 프로그램**을 제공하여  
두 버전을 동시에 사용하거나, 점진적으로 이전할 수 있도록 설계했습니다.

## 6.2 ROS1과 ROS2의 차이점 (개념 요약)

원래는 이 부분에 **비교 표**가 들어가지만,  
여기서는 표 대신 개념적으로 몇 가지 중요한 차이를 요약합니다.

- **통신 계층**
  - ROS1: TCPROS/UDPROS 기반, 자체 프로토콜
  - ROS2: DDS(DDSI-RTPS) 기반, 산업 표준 채택

- **아키텍처**
  - ROS1: 마스터 노드(roscore)에 의존하는 구조
  - ROS2: 탈중앙 분산 구조, DDS의 Dynamic Discovery 사용

- **실시간성**
  - ROS1: 실시간 제어를 직접 지원하지 않음
  - ROS2: RTOS, 실시간 통신, 메모리 관리 등을 고려한 설계

- **보안**
  - ROS1: 기본 보안 기능 부재
  - ROS2: DDS-Security 도입, SROS2 등 보안 강화

- **플랫폼**
  - ROS1: 주로 리눅스(Ubuntu) 중심
  - ROS2: 리눅스, 윈도우, macOS 및 임베디드 환경까지 확장

## 6.3 ROS2의 주요 특징

아래 항목들은 “ROS1과 비교했을 때 ROS2가 어떤 점에서 달라졌는가?”를 보여주는 핵심 특징들입니다.

### 1) Platforms

- 리눅스, 윈도우, macOS를 모두 지원합니다.
- 개발 환경은 맥/윈도우, 실제 로봇은 리눅스처럼 **크로스 플랫폼 개발**이 가능합니다.

### 2) Real-time

- ROS2는 **실시간성을 지원할 수 있는 구조**로 설계되었습니다.
- 단, 다음 전제를 만족해야 합니다.
  - 선별된 하드웨어
  - 실시간 지원 운영체제(RTOS 또는 튜닝된 Linux)
  - DDS의 RTPS와 같은 실시간 통신 프로토콜
  - 실시간 코딩 규칙을 따른 애플리케이션 코드

### 3) Security

- ROS2는 OMG에서 산업용으로 사용 중인 **DDS**를 도입하고,  
  그 위에 **DDS-Security**를 적용했습니다.
- 커뮤니티에서는 `SROS2` 프로젝트를 통해
  - 암호화
  - 인증
  - 접근 제어  
  등 보안 관련 기능을 강화하고 있습니다.

### 4) Communication

- DDS에서는 **IDL**을 사용해 메시지를 정의하고 직렬화하여,  
  더 쉽게·더 포괄적으로 메시지를 다룰 수 있습니다.
- 통신 프로토콜로 **RTPS**를 채택하여 실시간 데이터 전송을 보장하고,  
  임베디드 시스템에서도 사용 가능하도록 설계되었습니다.
- DDS는
  - 노드 간 자동 감지(Dynamic Discovery)
  - QoS 설정을 통한 통신 특성 조절  
  을 지원합니다.

### 5) Middleware interface (RMW)

- ROS2에서는 다양한 DDS 벤더를 사용 목적에 맞게 선택할 수 있도록  
  **RMW(Robot Middleware Wrapper)** 계층을 제공합니다.
- RMW는 여러 DDS 구현을 지원하기 위한 **추상화 인터페이스(API)** 로,  
  상위 레벨 코드는 DDS 벤더에 의존하지 않고 동일한 방식으로 동작합니다.

### 6) Node manager (Discovery)

- ROS2에서 노드는 DDS의 **Participant** 개념과 연결됩니다.
- DDS의 Dynamic Discovery를 활용해,
  - 네트워크 상의 노드를 자동으로 검색하고
  - 별도의 중앙 마스터 없이도 서로 연결할 수 있습니다.

### 7) Languages

- ROS2의 주력 언어는 **C++** 와 **Python**입니다.
- 그 외에도 버전에 따라
  - `rcljava`, `rclobjc`, `rclada`, `rclgo`, `rclnodejs` 등  
  다양한 언어 바인딩이 제공됩니다.

### 8) Build system

- ROS2는 **ament** 빌드 시스템을 사용합니다.
- `ament_cmake`를 통해 CMake 기반 C++ 빌드를 지원하고,  
  `ament_python`을 통해 Python 패키지 관리도 지원합니다.

### 9) Build tools

- **colcon**은 ROS2 패키지의 작성, 테스트, 빌드 등을 지원하는  
  CLI 타입의 통합 빌드 도구입니다.
- 여러 패키지를 한 번에, 의존성 순서를 고려해, 병렬로 빌드할 수 있어  
  작업 흐름이 크게 향상됩니다.

### 10) Build options

- 빌드 도구 변경과 함께 옵션에도 변화가 생겼습니다.
- 주요 특징
  - 여러 개의 독립된 워크스페이스 사용 가능
  - **격리 빌드(isolated build)** 지원 → 패키지별로 완전히 분리된 빌드
  - 개별 패키지를 빌드 후, 바로 해당 패키지만 사용 가능
  - **심볼릭 링크 설치**를 통해 개발 중 코드 변경을 빠르게 반영

### 11) Version control system

- **vcstool** 을 사용해 여러 리포지토리를 한 번에 관리할 수 있습니다.
- 예를 들어, 여러 Git 리포지토리를 하나의 워크스페이스에 가져올 때  
  `.repos` 파일과 `vcs import` / `vcs pull` 명령을 활용합니다.

### 12) Client library

- ROS2는 언어별로 **클라이언트 라이브러리**를 제공합니다.
  - C++: `rclcpp`
  - C: `rclc`
  - Python: `rclpy`
  - Java: `rcljava`
  - Objective-C: `rclobjc`
  - Ada: `rclada`
  - Go: `rclgo`
  - Node.js: `rclnodejs`
- 이를 통해 다양한 언어에서 ROS2 노드를 작성할 수 있습니다.

### 13) Lifecycle

- 각 노드의 현재 상태를 모니터링하고, 상태 전이를 제어할 수 있는  
  **Lifecycle Node** 개념을 제공합니다.
- 예:
  - `unconfigured → inactive → active → shutting down` 등의 상태 전이
- 이를 통해 ROS 시스템의 상태를 더 명확하게 관리하고,  
  안전한 초기화/종료 시퀀스를 구성할 수 있습니다.

### 14) Multiple nodes (Components)

- ROS2에서는 ROS1의 Nodelet 개념을 확장한 **Component** 를 사용합니다.
- 하나의 실행 파일(process) 안에서 **여러 노드**를 실행할 수 있어
  - 프로세스 수를 세분화하고
  - 프로세스 간 통신 오버헤드를 줄이며
  - 더 효율적인 응용 프로그램 구성이 가능합니다.

### 15) Threading model

- ROS2에서는 C++과 Python 모두에서 **더 세분화된 실행 모델(executor)** 을 제공합니다.
- 예:
  - `single-threaded executor`
  - `multi-threaded executor` (for `rclcpp`)
- 사용자는 executor API를 활용해 자신의 워크로드에 맞는 실행 모델을 정의할 수 있습니다.

### 16) Messages (Topic, Service, Action)

- ROS2에서는 DDS를 기반으로 Topic, Service, Action을 구현합니다.
- IDL을 사용하여 메시지 정의·직렬화를 수행함으로써  
  다양한 언어와 시스템 간 상호 운용성을 높이고 있습니다.

### 17) Command Line Interface

- ROS2는 풍부한 CLI 도구를 제공합니다.
  - 예: `ros2 topic`, `ros2 service`, `ros2 node`, `ros2 run`, `ros2 launch` 등
- 이를 통해 코드 작성 전에, 또는 디버깅 중에  
  **시스템 상태를 손쉽게 조회·제어**할 수 있습니다.

### 18) Launch

- `launch`는 사용자 지정 프로그램 실행을 관리하는 도구입니다.
- 단순 실행인 `run` 과 달리
  - 여러 노드 동시 실행
  - 파라미터 설정
  - 네임스페이스/리맵
  - 조건부 실행  
  등의 복잡한 구성을 지원합니다.
- Python(`.py`)과 XML 형식 모두를 지원하며,  
  사용자가 원하는 실행 구성을 코드로 명시할 수 있습니다.

### 19) Graph API

- 각 노드와 메시지(토픽, 서비스, 액션)는 고유한 이름을 가지고 매핑됩니다.
- Graph API를 통해
  - 현재 어떤 노드가 실행 중인지
  - 어떤 토픽을 발행/구독하는지  
  등 **통신 관계를 그래프 구조로 조회**할 수 있습니다.

### 20) Embedded Systems

- 임베디드 보드에서 직접 ROS2 프로그래밍을 수행하여,  
  펌웨어 수준에서 노드를 실행할 수도 있습니다.
- 이를 통해
  - 모터 제어
  - 센서 읽기  
  등 **실시간성이 중요한 부분을 분리**해 개발할 수 있습니다.

## 6.4 정리

- ROS2는 ROS1의 한계를 보완하기 위해 **통신, 실시간성, 보안, 플랫폼, 빌드 시스템** 등  
  거의 모든 층위에서 구조적 개선을 수행한 플랫폼입니다.
- 1~4장에서 다룬 ROS의 개념 위에,  
  이 장의 내용을 더해 보면 “왜 ROS2가 현대 로봇 소프트웨어의 기준이 되었는가”를  
  한층 더 입체적으로 이해할 수 있습니다.
