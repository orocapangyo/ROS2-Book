# 7장. ROS2와 DDS

## 7.1 ROS2와 DDS의 관계

ROS2에서 사용되는 메시지 통신 방식에는 **토픽(Topic), 서비스(Service), 액션(Action), 파라미터(Parameter)** 가 있습니다.  
ROS 전체에서 **퍼블리셔(Publisher)와 서브스크라이버(Subscriber)** 는 핵심 개념으로 자리 잡고 있죠.

ROS2는 이 통신을 구현하기 위해, OMG(Object Management Group)에서 표준화한  
**DDS(Data Distribution Service)** 의 프로토콜인 **DDSI-RTPS** 를 사용합니다.

### DDS 도입으로 인한 변화

- OMG의 **IDL(Interface Definition Language)** 을 사용하여 메시지 타입을 정의하고 직렬화할 수 있습니다.
- DDS의 중요 콘셉트인 **DCPS, DLRL** 내용을 반영한 DDSI-RTPS를 채택하여
  - 실시간 데이터 전송을 보장하고
  - 임베디드 시스템에서도 사용 가능하게 되었습니다.
- DDS를 통해 노드 간 **동적 검색(Dynamic Discovery)** 기능을 사용할 수 있습니다.
- 또한 노드 간 데이터 통신을 세부적으로 조정하는 **QoS(Quality of Service)** 를 설정할 수 있어,
  - TCP처럼 데이터 손실을 방지하여 **신뢰도 우선**
  - UDP처럼 재전송 부담을 줄이고 **속도 우선**  
  의 선택이 가능합니다.
- **DDS-Security** 도입으로 보안 측면에서도 큰 향상을 얻었습니다.

결과적으로, DDS 덕분에 개발자와 사용자는  
통신 미들웨어에 대한 구현 부담을 줄이고, **로봇 애플리케이션 로직에 더 집중**할 수 있게 되었습니다.

---

## 7.2 DDS란 무엇인가?

**DDS(Data Distribution Service)** 는 이름 그대로 “데이터 분산 서비스”를 제공하는  
데이터 통신용 미들웨어입니다.

- DDS는 **데이터를 중심으로 연결성을 가지는 미들웨어**입니다.
- DDS 사양을 만족하는 미들웨어 API와, 그에 대응하는 프로토콜(DDSI-RTPS)이  
  DDS의 실체라고 볼 수 있습니다.
- 네트워크 상의 여러 호스트가 DDS를 통해 통신하면서 데이터를 공유할 수 있으며,  
  ROS2는 이 계층을 통신 백본으로 활용하고 있습니다.

## 7.3 DDS의 주요 특징

### 7.3.1 Industry Standards

- DDS는 비영리 단체인 **OMG** 관리하에 정의된 산업 표준입니다.
- ROS2가 DDS를 도입함으로써
  - IoT
  - 자동차
  - 국방
  - 항공
  - 우주  
  등 더 넓은 산업 영역으로 확장할 수 있는 기반을 마련했습니다.

### 7.3.2 OS Independent

- DDS는 다양한 운영체제를 지원합니다.
- 따라서 리눅스, 윈도우, RTOS 등 다양한 환경에서 동작하는 ROS2 시스템에  
  매우 잘 맞는 통신 계층입니다.

### 7.3.3 Language Independent

- DDS는 미들웨어이기 때문에, **사용자 코드 레벨에서 특정 언어에 묶이지 않습니다.**
- ROS2에서는 이 특징을 살려 DDS를 **RMW** 로 추상화했고,  
  벤더별로 서로 다른 DDS 구현을 선택할 수 있습니다.

### 7.3.4 Transport on UDP/IP

- DDS 구현은 일반적으로 UDP/IP 위에서 동작하며,
  - 신뢰성 있는 멀티캐스트,
  - 특정 도메인 그룹에 대한 데이터 전송  
  등을 지원합니다.
- ROS2에서는 DDS Global Space에 등록된 토픽들에 대해  
  퍼블리시·서브스크라이브를 수행할 수 있습니다.

### 7.3.5 Data Centricity

- DDS는 이름 그대로 **데이터 중심(Data-Centric)** 입니다.
- DDS의 DCPS 개념은
  - “적절한 데이터를, 적절한 수신자에게, 효율적으로 전달하는 것”을 목표로 합니다.
- 미들웨어는
  - 어떤 데이터를
  - 어떤 형식으로
  - 어떻게 보내고
  - 어떻게 안전하게 보낼지  
  에 대한 기능을 제공하여, 데이터 중심 설계를 돕습니다.

### 7.3.6 Dynamic Discovery

- DDS는 **동적 검색(Dynamic Discovery)** 를 제공합니다.
- 응용 프로그램은 DDS를 통해
  - 어떤 도메인에 어떤 토픽이 존재하는지,
  - 어떤 노드가 발신/수신을 담당하는지  
  를 자동으로 알 수 있습니다.
- ROS2에서는 노드를 DDS의 **Participant** 로 취급하며,  
  동적 검색 기능을 이용해 자동으로 연결합니다.

### 7.3.7 Scalable Architecture

- DDS 아키텍처는
  - 소형 디바이스에서부터
  - 대규모 인프라, 국방, 항공, 우주 산업과 같은 초대형 시스템까지  
  확장할 수 있도록 설계되었습니다.
- 분산 시스템 개발을 단순화하고 편의성을 높여,  
  다수의 로봇, 다양한 IT 인프라, 데이터베이스, 클라우드와 연계되는  
  ROS2 시스템에 매우 적합합니다.

### 7.3.8 Interoperability

- DDS는 **상호 운용성(Interoperability)** 을 지원합니다.
- DDS 벤더를 교체하더라도, 혹은 서로 다른 벤더 간에도  
  통신이 가능하도록 설계되어 있습니다.
- 이것이 ROS2에서 **여러 DDS 벤더를 선택적으로 사용할 수 있는 기반**이 됩니다.

### 7.3.9 Quality of Service (QoS)

- 노드 간 DDS 통신 옵션을 **QoS** 로 설정할 수 있으며,  
  퍼블리셔/서브스크라이버 선언 시 사용합니다.
- QoS를 통해 예를 들어
  - 데이터 손실을 최대한 방지(신뢰도 우선)
  - 손실을 허용하되, 속도를 최우선(속도 우선)  
  과 같은 정책을 선택할 수 있습니다.
- 또한
  - 정해진 크기만큼 데이터를 보관할지,
  - 수신자가 생기기 이전의 데이터를 보관/폐기할지,
  - 정해진 주기 안에 수신되는 데이터만 유효로 볼지,
  - 정해진 주기로 노드/토픽의 생존 여부를 확인할지  
  등의 세부 동작도 설정할 수 있습니다.

### 7.3.10 Security

- DDS의 **DDS-Security** 를 적용함으로써,  
  ROS1의 약점이었던 보안 문제를 ROS2에서 크게 개선할 수 있었습니다.
- 암호화, 인증, 접근 제어 등을 미들웨어 레벨에서 지원합니다.

## 7.4 ROS2에서 DDS 활용하기

### 7.4.1 RMW와 상호 운용성 테스트

- ROS2에서는 `demo_nodes_cpp` 패키지에 포함된 `talker` / `listener` 예제를 실행하여  
  DDS 통신을 쉽게 테스트할 수 있습니다.
- `rqt_graph` 를 실행하면 각 노드와 토픽이 **그래프 형태로 시각화** 되는 것을 볼 수 있습니다.

기본 RMW 구현은 보통 `rmw_fastrtps_cpp` 가 사용되며,  
다른 DDS 구현을 사용하려면 `RMW_IMPLEMENTATION` 환경 변수를 지정한 후 노드를 실행합니다.

```bash
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp   # 예시
```

이 값을 다른 터미널에서 서로 다르게 설정한 후 통신을 시험해 보면,  
서로 다른 RMW(=서로 다른 DDS 벤더)를 사용해도 문제없이 통신되는 것을 확인할 수 있습니다.  
이는 앞에서 설명한 DDS의 **상호 운용성**을 실제로 체험하는 좋은 예입니다.

### 7.4.2 도메인(Domain)을 이용한 통신망 분리

- ROS2에서는 기본적으로 **UDP 멀티캐스트** 로 통신합니다.
- 별도의 설정을 하지 않으면, 동일 네트워크 상의 모든 노드가  
  하나의 큰 그래프로 연결될 수 있습니다.

이를 방지하기 위해서는

- ROS 네임스페이스를 활용하거나
- **Domain ID** 를 변경하는 방법을 사용할 수 있습니다.

```bash
export ROS_DOMAIN_ID=10
```

`ROS_DOMAIN_ID` 환경 변수를 다르게 지정하면,  
**도메인 ID가 같은 노드끼리만 서로 통신**하게 되어  
통신망을 논리적으로 분리할 수 있습니다.

### 7.4.3 QoS 설정 예시

- `Reliability` 설정이 `RELIABLE` 인 경우
  - 데이터 손실이 발생하더라도, TCP처럼 ACK를 확인하며 손실 데이터를 재전송합니다.
  - 데이터 유실은 거의 없지만, 재전송으로 인해 **일시적인 끊김**이 발생할 수 있습니다.

- `Reliability` 설정이 `BEST_EFFORT` 인 경우
  - 데이터 손실이 발생해도 재전송을 하지 않으므로,  
    끊김 없이 계속해서 수신을 이어갑니다.
  - 일부 데이터 손실을 허용해도 되는 센서 데이터(예: 카메라 영상) 등에는  
    `BEST_EFFORT` 가 적합할 수 있습니다.

손실이 있어도 큰 문제가 없는 상황이라면,  
오히려 `BEST_EFFORT` 를 사용하는 것이 전체 시스템 반응성을 높이는 데 유리할 수 있습니다.
