# 23. ROS2의 빌드 시스템과 빌드 툴

## 23.1 빌드 시스템과 빌드 툴

빌드 시스템은 단일 패키지를 빌드합니다. 빌드 툴과의 차이점은 전체를 대상으로 한다는 것입니다.
단일 패키지와 시스템 전체를 나누게 된 것은 의존성 때문.
코드의 재사용성을 위하여 패키지와 노드 단위로 구성되어 있고, 각 패키지는 다른 패키지와 상호 호환성을 위하여 의존성을 갖게된다.
패키지 간의 의존성 존재. 단일 패키지의 경우 빌드 시스템을 사용, 빌드 시스템은 단일 패키지에서 동작하는 것으로 생각하면 되며 의존성을 해결하고 빌드하여 실행 가능한 파일을 생성하는 것.

ROS에서는 수많은 패키지를 함께 빌드하여 실행시키는 구조이기 때문에 패키지 간의 종속성이 매우 얽혀있는 경우 의존성 실타래를 파악, 토폴로지 순서대로 빌드 필요. 이 때 빌드 툴을 사용한다.
빌드 툴은 각 패키지에 기술되어 있는 종속성 그래프를 해석, 각 패키지에 대한 특정 빌드 시스템을 호출.

## 23.2 빌드 시스템

ROS1에서는 catkin 빌드 시스템을 사용하지만, ROS2는 ament를 사용합니다. 이 두 빌드 시스템 모두 CMake로 만들어졌습니다.
CMake를 이용하는 이유는 ROS2는 멀티 플랫폼에서 빌드할 수 있게 하기 위함. 리눅스, 맥, 윈도우 계열도 지원 가능.
또한 다양한 IDE 에서 기본적으로 지원하며 쉽게 사용할 수 있다.
ROS1의 catkin과 ROS2의 ament가 다른 점은 파일 시스템에서 devel 공간을 사용하지 않는다는 것과 CMAKE_PREFIX_PATH이 아닌 AMENT_PREFIX_PATH와 같은 고유 환경설정을 사용할 수 있다는 것. 또한 파이썬 패키지 관리도 지원.

ament 사용으로 ROS2에 와서 파이썬 패키지가 독립을 이루었다. setuup.py파일의 모든 기능을 파이썬 모듈과 동등한 수준으로 개발할 수 있게 되었다.

## 23.3 빌드 툴

빌드 툴은 전체 패키지를 대상으로 하므로 하나의 패키지가 아닌 전체를 대상으로 합니다. 따라서 별도의 도구가 필요합니다. ROS2에서는 colcon(Collective Construction)이라는 빌드 툴을 사용합니다.
ROS2 ROS1 모두 지원하기 위하여 통합된 빌드 툴로써 기본 빌드 툴로 사용중에 있다.

## 23.4 패키지 생성

패키지 생성하는 방법은 1. 직접 패키지 폴더를 만들고 파일 시스템에 필수적인 파일 포함해 소스코드 작성. 2. ros2cli 명령어 사용. 3. ai 이용

ros2 pkg create 명령어를 통해 패키지의 기본 내용을 쉽게 생성할 수 있다.

ros2 pkg create (패키지명) --build-type (빌드타입) --dependencies (패키지1) ...
빌드 타입은 ament_cmake와 ament_python 중에서 선택하여 개발 언어를 정할 수 있습니다.

## 23.5 패키지 빌드

ROS2에서 특정 패키지 또는 전체 패키지를 빌드할 때는 colcon 빌드 도구를 사용합니다. colcon은 Python으로 작성된 통합 빌드 도구입니다. -package-select 옵션을 사용하여 특정 패키지를 빌드할 수 있습니다.
주료 옵션 --packages-select 특정 패키지 골라서 , --packages-up-to 는 의존성 패키지도 함께

버전 컨트롤 시스템 툴. vcs 명령어 이용. 다양한 리포지터리와 혼재된 버전 관리 시스템을 사용하더라도 ros2에서 사용 가능?

### 23.6.2 rosdep

package.xml 에 기술된 의존성 정보를 가지고 의존성 패키지들을 설치해주는 역할. 사용자가 직접 설치하는 것 보다 편하며, 협업 시 유용.

### 23.6.3 bloom

바이너리 패키지 관리 툴. 개발한 패키지를 apt 명령어로 사용할 수 있게 할 수 있다. 메타데이터를 생성하고, 바이너리 패키지를 빌드하는데 사용할 수 있도록 함. 개발이 모두 끝난 이후 이를 배포하고 유지 관리할 때 사용하게 된다.
