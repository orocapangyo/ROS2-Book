## 슬라이드 2. 패키지 구성 요소 및 역할

먼저 이번 예제 패키지의 전체 구성을 보겠습니다.

이 패키지는 총 4개의 노드로 구성됩니다.  
첫 번째는 `argument_node`입니다. 이 노드는 변수 a와 b를 생성하고 Topic으로 발행합니다.

두 번째는 `operator_node`입니다. 이 노드는 덧셈, 뺄셈, 곱셈, 나눗셈 같은 사칙연산을 Service 방식으로 요청합니다.

세 번째는 `calculator_node`입니다. 이 노드는 실제 연산을 수행하는 중심 노드입니다. Topic으로 전달받은 값을 바탕으로 Service 요청에 따라 계산을 수행합니다.

네 번째는 `checker_node`입니다. 이 노드는 Action 클라이언트로 동작하면서 계산 결과의 누적값을 확인합니다. 특정 임계값을 넘으면 작업을 종료하는 역할을 합니다.

이처럼 하나의 패키지 안에서 Topic, Service, Action이 함께 사용되며, 실제 ROS 2 시스템에 가까운 구조를 실습할 수 있습니다.

## 슬라이드 3. 노드별 상세 역할

이번에는 각 노드의 역할을 조금 더 구체적으로 보겠습니다.

`argument_node`는 랜덤한 두 개의 변수 a와 b를 생성합니다.  
그리고 이 값을 Topic으로 계속 발행합니다. 즉, 계산에 필요한 입력값을 제공하는 노드입니다.

`operator_node`는 어떤 연산을 수행할지 결정합니다.  
예를 들어 더하기, 빼기, 곱하기, 나누기 중 하나를 선택해서 Service 요청으로 전달합니다.

`calculator_node`는 이 시스템의 핵심입니다.  
`argument_node`에서 받은 값과 `operator_node`에서 요청한 연산자를 조합해서 실제 계산을 수행합니다.

마지막으로 `checker_node`는 Action 클라이언트로서 누적 결과를 확인합니다.  
계산이 반복되는 동안 현재 결과를 피드백으로 받고, 결과가 특정 기준값을 넘으면 종료 조건을 판단합니다.

즉, 이 예제는 단순 계산기이지만 ROS 2의 핵심 통신 구조를 모두 포함하고 있습니다.

## 슬라이드 4. GitHub 저장소와 패키지 구조

이번 예제 코드는 GitHub 저장소에서 확인할 수 있습니다.

저장소에는 인터페이스 예제 코드와 Python, C++ 예제 코드가 함께 포함되어 있습니다.

`package.xml`은 이 패키지가 어떤 의존성을 가지는지 선언하는 파일입니다.  
예를 들어 rclpy나 custom interface 패키지처럼 다른 패키지를 사용한다면 이 파일에 의존성을 명시해야 합니다.

`setup.py`는 Python 패키지를 ROS 2에서 실행 가능하도록 설정하는 파일입니다.  
특히 어떤 Python 파일을 실행 명령어로 연결할지, 어떤 launch 파일과 yaml 파일을 설치할지 정의합니다.

## 슬라이드 5. package.xml 의존성 이해

ROS 2 패키지를 빌드할 때 가장 중요한 부분 중 하나가 의존성 관리입니다.

`package.xml`은 이 패키지가 빌드되고 실행되기 위해 필요한 다른 패키지들을 선언합니다.  
만약 custom message, service, action interface를 사용한다면 해당 인터페이스 패키지가 먼저 빌드되어 있어야 합니다.

예를 들어 계산 요청이나 액션 결과를 별도의 인터페이스로 정의했다면, 그 인터페이스 패키지를 dependency로 추가해야 합니다.

의존성이 정확히 선언되지 않으면 빌드 단계에서 에러가 발생하거나, 실행 시 필요한 타입을 찾지 못하는 문제가 생길 수 있습니다.

따라서 ROS 2 패키지를 설계할 때는 코드 작성뿐 아니라, `package.xml`에서 의존성을 정확히 선언하는 것이 매우 중요합니다.

## 슬라이드 6. setup.py의 주요 역할

다음은 `setup.py`입니다.

ROS 2 Python 패키지에서 `setup.py`는 실행 구조를 정의하는 핵심 파일입니다.

특히 중요한 부분은 두 가지입니다.  
첫 번째는 `entry_points`입니다.  
여기서는 각 노드의 실행 이름과 실제 Python 파일의 `main` 함수를 연결합니다.

예를 들어 `ros2 run` 명령으로 `calculator_node`를 실행하면, setup.py에 등록된 entry point를 통해 해당 Python 파일의 main 함수가 호출됩니다.

두 번째는 `data_files`입니다.  
여기서는 launch 파일이나 yaml 파라미터 파일처럼 실행에 필요한 리소스를 install 폴더로 복사하도록 지정합니다.

즉, `setup.py`는 Python 코드와 ROS 2 실행 시스템을 연결해주는 다리 역할을 합니다.

---

## 슬라이드 7. data_files와 설치 경로

ROS 2에서 launch 파일이나 yaml 파일은 단순히 소스 폴더에만 있다고 해서 자동으로 인식되지 않습니다.

빌드 후 실행하려면 이 파일들이 install 폴더의 적절한 위치에 복사되어야 합니다.  
이 작업을 담당하는 것이 setup.py의 `data_files` 설정입니다.

보통 launch 파일은 `share/<package_name>/launch` 경로에 설치되고, yaml 파일은 `share/<package_name>/config` 같은 경로에 설치됩니다.

또한 Python에서는 `glob` 함수를 사용해서 `*.launch.py`나 `*.yaml`처럼 특정 확장자를 가진 파일들을 자동으로 찾아 리스트에 포함할 수 있습니다.

이 설정을 잘 해두면 나중에 `ros2 launch` 명령을 사용할 때 ROS 2가 해당 launch 파일을 정확히 찾을 수 있습니다.

## 슬라이드 8. 패키지 빌드 명령어

이제 패키지를 빌드하는 과정을 보겠습니다.

먼저 의존성 설치는 `rosdep install --from-paths src --ignore-src -r -y` 명령으로 진행할 수 있습니다.  
이 명령은 src 폴더 아래의 패키지를 기준으로 필요한 의존성을 확인하고 설치합니다.

그다음 전체 워크스페이스를 빌드할 때는 `colcon build` 명령을 사용합니다.

만약 특정 패키지만 선택해서 빌드하고 싶다면 `colcon build --packages-select <package_name>` 명령을 사용합니다.

빌드가 완료되면 Python 스크립트와 실행 파일, launch 파일, yaml 파일 등이 install 폴더 아래에 배치됩니다.

ROS 2에서는 이 install 공간을 기준으로 패키지를 실행하기 때문에, 빌드 후에는 반드시 환경 설정을 source 해주는 것도 중요합니다.

## 슬라이드 9. 노드 실행 방식

빌드가 완료되면 각 노드는 `ros2 run` 명령으로 실행할 수 있습니다.

기본 형식은 `ros2 run <package_name> <executable_name>`입니다.

여기서 `<executable_name>`은 setup.py의 entry_points에 등록한 이름입니다.  
따라서 Python 파일 이름과 실행 이름이 반드시 같을 필요는 없지만, setup.py에 정확히 연결되어 있어야 합니다.

예를 들어 calculator 노드를 실행하려면 패키지 이름과 실행 이름을 입력해서 실행합니다.

개별 노드 실행 방식은 디버깅할 때 매우 유용합니다.  
각 노드를 하나씩 실행하면서 Topic이 제대로 발행되는지, Service 요청이 정상적으로 처리되는지, Action 결과가 잘 전달되는지 확인할 수 있습니다.

## 슬라이드 10. 전체 노드 실행 흐름

이제 네 개의 노드가 어떻게 함께 동작하는지 흐름을 정리해보겠습니다.

먼저 `argument_node`가 랜덤 숫자 a와 b를 생성합니다.  
이 값은 Topic으로 발행됩니다.

그다음 `calculator_node`가 이 Topic을 구독해서 a와 b 값을 받습니다.

`operator_node`는 어떤 연산을 수행할지 Service 요청을 보냅니다.  
예를 들어 덧셈을 요청하면 calculator node는 현재 받은 a와 b 값을 더해서 결과를 반환합니다.

이후 `checker_node`는 Action 방식으로 누적 결과를 확인합니다.  
계산이 진행되는 동안 피드백을 받고, 총합이 특정 임계값을 초과하면 작업을 종료합니다.

이 구조를 통해 ROS 2에서 Topic, Service, Action이 각각 어떤 상황에 적합한지 자연스럽게 이해할 수 있습니다.

## 슬라이드 11. Topic, Service, Action 비교

이번 예제에서 중요한 것은 세 가지 통신 방식의 차이를 이해하는 것입니다.

Topic은 지속적인 데이터 스트림에 적합합니다.  
예를 들어 센서 데이터, 위치 정보, 상태 정보처럼 계속 변하는 값을 전달할 때 사용합니다.  
이번 예제에서는 a와 b 값이 Topic으로 전달됩니다.

Service는 요청과 응답이 명확한 작업에 적합합니다.  
예를 들어 “이 두 값을 더해줘”처럼 요청하면 바로 결과를 돌려받는 구조입니다.  
이번 예제에서는 연산 요청이 Service로 처리됩니다.

Action은 시간이 오래 걸리는 작업에 적합합니다.  
작업을 요청한 뒤 중간 진행 상황을 피드백으로 받고, 마지막에 결과를 받습니다.  
이번 예제에서는 누적 계산 결과를 확인하고 종료 조건을 판단하는 데 Action이 사용됩니다.

## 슬라이드 12. 런치 파일의 필요성

개별 노드를 하나씩 실행하는 방식은 학습이나 디버깅에는 좋지만, 실제 시스템 운영에는 불편합니다.

노드가 4개만 되어도 터미널을 여러 개 열어야 하고, 실행 순서나 파라미터 설정도 매번 신경 써야 합니다.

이 문제를 해결하기 위해 ROS 2에서는 launch 파일을 사용합니다.

launch 파일을 사용하면 여러 노드를 한 번에 실행할 수 있습니다.  
또한 각 노드에 필요한 파라미터, 실행 인자, 조건부 실행 등을 함께 관리할 수 있습니다.

즉, launch 파일은 ROS 2 시스템을 오케스트레이션하는 도구입니다.  
실제 프로젝트에서는 launch 파일을 잘 설계하는 것이 패키지 구조만큼 중요합니다.

## 슬라이드 13. YAML 파라미터 활용

ROS 2에서는 노드의 설정값을 YAML 파일로 관리할 수 있습니다.

예를 들어 checker node에서 임계값을 50으로 할지, 100으로 할지 코드에 직접 작성하지 않고 YAML 파일로 분리할 수 있습니다.

이렇게 하면 코드를 수정하지 않고도 실행 조건을 바꿀 수 있습니다.

또한 여러 환경에서 다른 설정값을 적용하기도 쉬워집니다.  
개발 환경에서는 낮은 임계값을 사용하고, 실제 테스트 환경에서는 더 높은 임계값을 사용할 수 있습니다.

따라서 파라미터는 코드의 유연성과 재사용성을 높여주는 중요한 요소입니다.
