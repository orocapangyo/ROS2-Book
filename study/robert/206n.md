## 슬라이드 2. 토픽 통신 기본 개념

먼저 토픽 통신의 기본 개념부터 살펴보겠습니다.

토픽은 ROS 2에서 메시지를 주고받기 위한 대표적인 통신 방식입니다.  
데이터를 보내는 쪽을 퍼블리셔, 즉 Publisher라고 하고, 데이터를 받는 쪽을 서브스크라이버, 즉 Subscriber라고 합니다.

토픽 통신은 비동기식입니다.  
퍼블리셔가 메시지를 보낼 때 서브스크라이버가 반드시 동시에 대기하고 있어야 하는 것은 아닙니다.

또한 기본적으로 단방향 통신입니다.  
퍼블리셔가 메시지를 발행하면 서브스크라이버가 그 메시지를 받아 처리하는 구조입니다.

일반적으로는 일대일 구조로 설명하지만, 실제로는 하나의 퍼블리셔가 여러 서브스크라이버에게 데이터를 전달하는 일대다 구조도 가능합니다.

이 때문에 토픽은 센서 데이터, 상태 정보, 위치 정보처럼 지속적으로 흘러가는 데이터를 처리할 때 매우 많이 사용됩니다.

## 슬라이드 3. 퍼블리셔 노드의 역할

이번 예제에서 퍼블리셔 역할을 하는 노드는 `argument_node`입니다.

`argument_node`는 계산에 필요한 두 개의 값, 즉 a와 b를 생성합니다.  
이 값들은 랜덤 숫자로 만들어지고, 토픽을 통해 다른 노드로 전달됩니다.

즉, ROS 2 시스템에서는 각 노드가 하나의 역할을 명확히 갖는 것이 좋습니다.  
입력값을 만드는 노드, 계산을 수행하는 노드, 결과를 확인하는 노드를 분리하면 시스템 구조가 훨씬 명확해집니다.

이번 슬라이드에서는 퍼블리셔가 단순히 데이터를 보내는 노드라는 점을 이해하시면 됩니다.

## 슬라이드 5. 퍼블리셔 노드 초기화와 파라미터

## 슬라이드 6. QoS 설정의 의미

ROS 2 토픽 통신에서 중요한 개념 중 하나가 QoS입니다.

QoS는 Quality of Service의 약자로, 메시지를 어떤 방식으로 전달할지 결정하는 통신 품질 설정입니다.

예를 들어 depth는 메시지를 몇 개까지 보관할지 정하는 값입니다.  
depth가 10이면 최근 메시지 10개까지 저장할 수 있다는 의미입니다.

reliability를 reliable로 설정하면 메시지 전달의 신뢰성을 높입니다.  
durability를 transient local로 설정하면 늦게 접속한 구독자도 이전 메시지를 받을 수 있는 구조를 만들 수 있습니다.

또한 history는 메시지 보관 방식을 정하고, lifespan은 메시지가 유효한 시간을 설정합니다.

즉, QoS는 단순한 옵션이 아니라 ROS 2 통신의 안정성과 실시간성을 조정하는 중요한 설정입니다.

## 슬라이드 8. 서브스크라이버 노드의 역할

다음은 서브스크라이버 노드입니다.

이번 예제에서 서브스크라이버 역할을 하는 노드는 `calculator_node`입니다.

`calculator_node`는 `argument_node`가 발행한 `message_arithmetic` 토픽을 구독합니다.  
그리고 토픽으로 전달받은 `timestamp`, `a`, `b` 값을 읽어서 내부 변수에 저장합니다.

중요한 점은 서브스크라이버가 메시지를 직접 요청하지 않는다는 것입니다.  
토픽에 메시지가 발행되면, 등록된 콜백 함수가 자동으로 호출되어 메시지를 처리합니다.

즉, 서브스크라이버는 이벤트 기반으로 동작합니다.  
메시지가 도착하면 콜백 함수가 실행되고, 메시지가 없으면 대기합니다.

이 구조는 센서 데이터처럼 계속 들어오는 정보를 처리할 때 매우 적합합니다.

## 슬라이드 10. 콜백 함수 구현

서브스크라이버에서 가장 중요한 부분은 콜백 함수입니다.

토픽 메시지가 수신되면 이 함수가 자동으로 호출됩니다.  
함수 안에서는 수신한 메시지에서 `timestamp`, `a`, `b` 값을 꺼냅니다.

그리고 a와 b 값을 노드의 멤버 변수에 저장합니다.

또한 로그를 출력해서 어떤 메시지가 들어왔는지 확인할 수 있습니다.

콜백 함수는 ROS 2 프로그래밍에서 매우 핵심적인 개념입니다.  
토픽뿐 아니라 서비스, 액션, 타이머 등 여러 기능이 콜백 기반으로 동작하기 때문입니다.

## 슬라이드 11. 콜백 그룹과 Executor

ROS 2에서는 콜백을 어떻게 실행할지 제어하기 위해 콜백 그룹과 Executor를 사용합니다.

기본적으로 `SingleThreadedExecutor`는 한 번에 하나의 콜백만 실행합니다.  
즉, 현재 실행 중인 콜백이 끝나야 다음 콜백이 실행됩니다.

반면 여러 콜백을 병렬로 처리해야 하는 경우에는 `MultiThreadedExecutor`를 사용할 수 있습니다.

이때 `MultiThreadedCallbackGroup`과 함께 사용하면 여러 콜백 함수가 동시에 실행될 수 있습니다.

예를 들어 하나의 노드가 토픽도 받고, 서비스도 처리하고, 액션도 수행해야 한다면 단일 스레드만으로는 병목이 생길 수 있습니다.

이런 경우 멀티 스레드 구조를 사용하면 응답성과 처리 효율을 높일 수 있습니다.

## 슬라이드 12. setup.py 엔트리 포인트

ROS 2 Python 패키지에서 노드를 실행하려면 `setup.py`의 엔트리 포인트 설정이 필요합니다.

엔트리 포인트는 `ros2 run` 명령어와 실제 Python 코드의 `main` 함수를 연결해주는 역할을 합니다.

이번 예제에서는 `argument` 노드와 `calculator` 노드가 각각 엔트리 포인트로 등록됩니다.

예를 들어 `argument`라는 실행 이름을 입력하면, 해당 패키지 안의 `argument.py` 파일에 있는 `main` 함수가 실행됩니다.

마찬가지로 `calculator` 실행 이름은 `main.py`의 `main` 함수와 연결됩니다.

즉, Python 파일을 만들기만 해서는 ROS 2 명령어로 실행할 수 없습니다.  
반드시 setup.py에 실행 진입점을 등록해야 합니다.

## 슬라이드 13. argument 노드 실행 흐름

이제 `argument_node`의 실행 흐름을 보겠습니다.

먼저 `rclpy.init()`을 호출해서 ROS 2 Python 클라이언트 라이브러리를 초기화합니다.

그다음 `Argument` 클래스의 인스턴스를 생성합니다.  
이 객체가 바로 퍼블리셔 노드입니다.

이후 `rclpy.spin()`을 호출해서 노드를 계속 실행 상태로 유지합니다.  
spin이 동작하는 동안 타이머 콜백이 실행되고, 1초마다 메시지가 발행됩니다.

사용자가 Ctrl+C를 누르면 `KeyboardInterrupt`가 발생합니다.  
이때 종료 메시지를 출력하고 `rclpy.shutdown()`을 호출해서 ROS 2 시스템을 정상적으로 종료합니다.

이 흐름은 ROS 2 Python 노드 실행의 가장 기본적인 패턴입니다.

## 슬라이드 14. calculator 노드 실행 흐름

`calculator_node`는 조금 더 복잡합니다.

이 노드는 서브스크라이버 역할뿐 아니라 다른 기능도 함께 수행할 수 있기 때문에 멀티 스레드 Executor를 사용합니다.

먼저 노드 객체를 생성한 뒤, `MultiThreadedExecutor`를 생성합니다.  
스레드 수를 따로 지정하지 않으면 CPU 수에 맞춰 자동으로 설정됩니다.

그다음 `executor.add_node(node)`를 통해 노드를 Executor에 등록합니다.  
그리고 `executor.spin()`을 호출하면 등록된 노드의 콜백들이 실행됩니다.

종료 시에는 `executor.shutdown()`을 호출하고, 마지막으로 `rclpy.shutdown()`을 실행합니다.

자료에 따르면 calculator 노드에는 액션 서버도 포함되어 있으므로, 종료 시 액션 서버 정리 함수도 함께 실행됩니다.

이처럼 여러 기능을 가진 노드에서는 Executor 구조를 이해하는 것이 중요합니다.

마지막으로 메시지 발행 함수를 작성하고 타이머를 통해 주기적으로 호출합니다.

서브스크라이버도 비슷합니다.  
먼저 노드를 설정하고 QoS를 정의합니다.  
그다음 `create_subscription`으로 구독자를 생성하고, 메시지를 처리할 콜백 함수를 작성합니다.

정리하면 퍼블리셔는 “메시지를 만들어서 보내는 구조”이고, 서브스크라이버는 “메시지를 받아서 처리하는 구조”입니다.

ROS 2 토픽 프로그래밍은 이 두 구조를 정확히 이해하는 것에서 시작합니다.
