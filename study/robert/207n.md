## 슬라이드 2. 서비스 통신 개요

먼저 서비스 통신의 기본 개념부터 보겠습니다.

서비스는 Request-Response, 즉 요청-응답 구조입니다.  
서비스 클라이언트는 서비스를 요청하는 역할을 하고, 서비스 서버는 요청을 받아 처리한 뒤 응답을 반환하는 역할을 합니다.

예를 들어 클라이언트가 “두 숫자를 더해줘”라고 요청하면, 서버가 계산을 수행하고 “결과는 10입니다”라고 응답하는 방식입니다.

토픽과 비교하면 서비스는 지속적인 데이터 흐름보다는 특정 작업을 한 번 요청하고 결과를 받을 때 적합합니다.

따라서 센서 데이터처럼 계속 흘러가는 정보에는 토픽을 사용하고, 계산 요청이나 상태 확인처럼 명확한 응답이 필요한 작업에는 서비스를 사용하는 것이 좋습니다.

## 슬라이드 3. 서비스와 액션의 차이

서비스와 액션은 모두 요청을 보낸다는 점에서는 비슷합니다.  
하지만 동작 방식에는 중요한 차이가 있습니다.

서비스는 요청을 보내고 응답을 받으면 끝나는 단순한 구조입니다.  
중간 진행 상황을 알려주지 않고, 요청을 취소하는 기능도 기본적으로 없습니다.

반면 액션은 요청을 보낸 뒤 중간 피드백을 받을 수 있고, 최종 결과도 따로 받을 수 있습니다.  
또한 작업 중간에 취소하는 기능도 제공합니다.

예를 들어 단순 계산처럼 즉시 결과가 나오는 작업은 서비스가 적합합니다.  
하지만 로봇이 목적지까지 이동하는 것처럼 시간이 오래 걸리고 진행 상황을 알아야 하는 작업은 액션이 더 적합합니다.

이번 강의에서는 그중 서비스 구조를 Python 코드로 구현하는 방법을 집중적으로 살펴보겠습니다.

## 슬라이드 5. 서비스 메시지 인터페이스 정의

서비스를 사용하려면 먼저 요청과 응답의 형식을 정의해야 합니다.

이번 예제에서는 `ROS2_message_operator.srv` 파일을 사용합니다.

서비스 인터페이스 파일은 Request와 Response 영역으로 나뉩니다.  
중간에 `---` 구분선을 기준으로 위쪽은 요청 데이터, 아래쪽은 응답 데이터입니다.

요청에는 `a`, `b`, `operator` 값이 들어갑니다.  
`a`와 `b`는 계산에 사용할 정수이고, `operator`는 어떤 연산을 할지 나타냅니다.

예를 들어 operator 값이 1이면 덧셈, 2이면 뺄셈, 3이면 곱셈, 4이면 나눗셈으로 처리할 수 있습니다.

응답에는 `result`가 들어갑니다.  
서버는 계산 결과를 `float32` 타입의 result에 담아 클라이언트에게 반환합니다.

이처럼 `.srv` 파일은 서비스 클라이언트와 서버가 주고받을 데이터의 약속입니다.

## 슬라이드 6. 서비스 서버 생성

이제 서비스 서버 코드를 보겠습니다.

서비스 서버는 `create_service` 함수를 사용해서 생성합니다.

이 함수에는 서비스 메시지 타입, 서비스 이름, 콜백 함수, 그리고 필요하다면 콜백 그룹이 들어갑니다.

이번 예제에서는 `ROS2MessageOperator` 타입을 사용하고, 서비스 이름은 `ROS2_message_operator`입니다.

클라이언트도 같은 서비스 타입과 같은 서비스 이름을 사용해야 정상적으로 연결됩니다.

서비스 요청이 들어오면 `get_service_operator` 콜백 함수가 실행됩니다.

즉, 서비스 서버는 항상 대기하고 있다가 클라이언트의 요청이 들어오는 순간 콜백 함수를 통해 작업을 처리하는 구조입니다.

## 슬라이드 7. 서비스 서버 콜백 함수

서비스 서버에서 가장 중요한 부분은 콜백 함수입니다.

이번 예제의 콜백 함수는 `get_service_operator`입니다.

먼저 요청으로 들어온 `request.operator` 값을 확인합니다.  
이 값에 따라 어떤 연산을 수행할지 결정합니다.

operator 값이 1이면 덧셈 함수를 호출합니다.  
2이면 뺄셈, 3이면 곱셈, 4이면 나눗셈 함수를 호출합니다.

나눗셈의 경우에는 특별히 0으로 나누는 상황을 처리해야 합니다.  
0으로 나누면 프로그램 오류가 발생할 수 있기 때문에 반드시 예외 처리를 넣어야 합니다.

또한 정의되지 않은 operator 값이 들어왔을 때를 대비해 else 처리도 포함합니다.

마지막으로 계산 결과를 `response.result`에 넣고 반환합니다.  
이 반환값이 클라이언트에게 전달되는 서비스 응답입니다.

## 슬라이드 8. MultiThreadedExecutor 사용 이유

이번 서비스 서버 노드는 단순히 서비스만 처리하는 구조가 아닐 수 있습니다.

자료에서는 `calculate_operator` 노드가 토픽, 서비스, 액션 등 여러 작업을 처리하기 때문에 `MultiThreadedExecutor`를 사용한다고 설명합니다.

`SingleThreadedExecutor`는 하나의 작업을 순차적으로 처리합니다.  
따라서 하나의 콜백이 실행되는 동안 다른 콜백은 기다려야 합니다.

반면 `MultiThreadedExecutor`는 여러 작업을 동시에 처리할 수 있습니다.

예를 들어 토픽 메시지도 받아야 하고, 서비스 요청도 처리해야 하고, 액션 피드백도 관리해야 한다면 단일 스레드만으로는 응답성이 떨어질 수 있습니다.

이런 경우 MultiThreadedExecutor를 사용하면 여러 콜백을 병렬로 처리할 수 있어 복잡한 노드에 더 적합합니다.

## 슬라이드 9. 서비스 클라이언트 생성

이번에는 서비스 클라이언트 코드를 보겠습니다.

서비스 클라이언트는 `create_client` 함수를 사용해서 생성합니다.

이 함수에는 서비스 메시지 타입과 서비스 이름이 들어갑니다.

서버와 클라이언트가 통신하려면 두 가지가 반드시 일치해야 합니다.  
첫째, 서비스 메시지 타입이 같아야 합니다.  
둘째, 서비스 이름이 같아야 합니다.

이번 예제에서는 서버와 동일하게 `ROS2MessageOperator` 타입을 사용하고, 서비스 이름도 `ROS2_message_operator`로 설정합니다.

클라이언트는 이 객체를 통해 서버에게 요청을 보낼 수 있습니다.

즉, create_client는 서비스 서버와 연결하기 위한 클라이언트 통신 객체를 만드는 과정입니다.

## 슬라이드 10. 서비스 가용성 확인

서비스 클라이언트가 요청을 보내기 전에 반드시 확인해야 할 것이 있습니다.  
바로 서비스 서버가 준비되어 있는지 여부입니다.

이를 위해 `wait_for_service` 함수를 사용합니다.

예제에서는 0.2초 간격으로 서비스가 준비되었는지 확인합니다.  
만약 아직 서버가 준비되지 않았다면 “서비스가 준비되지 않았습니다”라는 로그를 출력합니다.

이 과정이 필요한 이유는 클라이언트가 서버보다 먼저 실행될 수 있기 때문입니다.

서버가 아직 실행되지 않았는데 클라이언트가 바로 요청을 보내면 실패할 수 있습니다.

따라서 실제 ROS 2 시스템에서는 서비스 요청 전에 wait_for_service로 서버의 존재를 확인하는 습관이 중요합니다.

## 슬라이드 11. 서비스 요청 생성과 비동기 호출

서비스가 준비되면 이제 요청 객체를 생성합니다.

서비스 요청은 `ROS2MessageOperator.Request()` 형태로 만듭니다.

그다음 요청 객체에 필요한 값을 넣습니다.  
이번 예제에서는 랜덤으로 선택된 연산자 값을 `request.operator`에 저장합니다.

그리고 `call_async` 함수를 사용해서 서버에 요청을 보냅니다.

여기서 중요한 점은 비동기 호출이라는 것입니다.

비동기 호출은 요청을 보낸 뒤 응답이 올 때까지 프로그램 전체가 멈추지 않습니다.  
요청을 보내놓고 다른 작업을 계속 수행할 수 있습니다.

서비스 응답은 future 객체를 통해 나중에 확인합니다.

이 방식은 ROS 2처럼 여러 콜백과 이벤트가 동시에 발생하는 환경에서 매우 유용합니다.

## 슬라이드 12. 동기 호출과 비동기 호출 비교

서비스 호출 방식에는 동기 방식과 비동기 방식이 있습니다.

동기 호출은 요청을 보낸 뒤 응답이 올 때까지 기다립니다.  
구조는 단순하지만, 응답이 늦어지면 그동안 다른 작업을 하지 못합니다.

반면 비동기 호출은 요청을 보낸 뒤에도 프로그램이 계속 동작합니다.  
응답이 도착하면 future 객체를 통해 결과를 확인합니다.

이번 예제에서는 `call_async`를 사용합니다.  
ROS 2 노드가 토픽, 서비스, 액션 등 여러 작업을 함께 처리해야 하는 경우에는 비동기 호출이 더 적합합니다.

또한 비동기 통신에서는 `spin_once`가 중요합니다.  
`spin_once`는 ROS 2에서 대기 중인 작업을 한 번 처리하는 함수입니다.

즉, 비동기 요청을 보낸 뒤 spin_once를 반복적으로 호출하면서 응답이 도착했는지 확인하는 구조로 동작합니다.

---

## 슬라이드 13. 서비스 응답 처리

비동기 요청을 보낸 뒤에는 future 객체를 통해 응답이 완료되었는지 확인합니다.

`self.future.done()`이 True가 되면 응답 처리가 완료된 상태입니다.

이때 `future.result()`를 호출해서 서버가 보낸 응답을 가져옵니다.

하지만 서비스 호출 중 오류가 발생할 수 있기 때문에 try-except 구조로 감싸는 것이 좋습니다.

예외가 발생하면 로그로 서비스 호출 실패를 출력합니다.

정상적으로 응답이 오면 `response.result` 값을 읽어서 결과를 출력합니다.

그리고 예제에서는 `user_trigger` 값을 False로 바꿔 다음 입력을 기다리는 상태로 전환합니다.

이렇게 하면 사용자가 다시 요청을 보낼 때까지 클라이언트가 대기하게 됩니다.

## 슬라이드 14. 서비스 클라이언트 메인 실행 흐름

서비스 클라이언트의 전체 실행 흐름을 정리해보겠습니다.

먼저 `rclpy.init()`으로 ROS 2 Python 환경을 초기화합니다.

그다음 operator 노드를 생성합니다.

이후 `ros_ok()` 루프 안에서 계속 동작합니다.

사용자 입력이나 특정 조건에 의해 `user_trigger`가 True가 되면 서비스 요청을 보냅니다.

그다음 `spin_once()`를 호출해서 ROS 2 관련 작업을 한 번 처리합니다.

응답이 도착하면 결과를 출력하고, `user_trigger`를 False로 바꾼 뒤 다시 대기 상태로 돌아갑니다.

사용자가 Ctrl+C를 누르면 KeyboardInterrupt가 발생하고, 노드를 종료한 뒤 `rclpy.shutdown()`으로 ROS 2를 정리합니다.

이 흐름이 비동기 서비스 클라이언트 구현의 핵심 구조입니다.

## 슬라이드 15. 예외 처리와 NOQA 주석

코드를 작성하다 보면 예외 처리와 코드 검사 관련 요소도 중요합니다.

먼저 `# NOQA` 주석은 “No Quality Assurance”의 약자입니다.  
특정 라인에 대해 코드 검사 도구가 경고를 표시하지 않도록 하는 역할을 합니다.

예를 들어 사용하지 않는 변수처럼 의도된 코드인데도 pylint 같은 도구가 경고를 표시할 수 있습니다.  
이때 `# NOQA`를 붙이면 해당 라인은 검사에서 제외됩니다.

다음으로 `try-except-else-finally` 구조를 보겠습니다.

`try` 블록에는 예외가 발생할 수 있는 코드를 작성합니다.  
`except` 블록은 예외가 발생했을 때 실행됩니다.  
`else` 블록은 예외가 발생하지 않았을 때 실행됩니다.  
`finally` 블록은 예외 발생 여부와 관계없이 항상 실행됩니다.

서비스 통신에서는 네트워크 상태, 서버 미실행, 잘못된 요청 등 다양한 오류가 생길 수 있으므로 예외 처리가 중요합니다.

## 슬라이드 16. setup.py 엔트리 포인트

ROS 2 Python 패키지에서 노드를 `ros2 run` 명령으로 실행하려면 setup.py에 엔트리 포인트를 등록해야 합니다.

엔트리 포인트는 실행 이름과 실제 Python 코드의 main 함수를 연결하는 설정입니다.

예제에서는 `operator`라는 실행 이름이 `ros2_message_operator.operator.main:main`과 연결됩니다.

즉, 사용자가 터미널에서 `ros2 run` 명령으로 operator를 실행하면, setup.py에 등록된 main 함수가 호출됩니다.

이 설정이 없으면 Python 파일이 존재하더라도 ROS 2 명령어로 바로 실행할 수 없습니다.

따라서 Python 기반 ROS 2 패키지를 만들 때는 코드 작성뿐만 아니라 setup.py 설정도 반드시 확인해야 합니다.
