# ROS 2 액션 프로그래밍 (Python)

ROS 2에서 액션 프로그래밍은 긴 시간 동안 실행되는 작업을 위한 통신 방식으로, 클라이언트가 요청하면 서버는 **피드백**을 제공하며 작업을 완료하고 **결과**를 반환한다. 이는 로봇의 움직임과 같이 시간이 오래 걸리는 작업에 효과적이다.

---

## 1. 액션 프로그래밍의 개념 및 필요성

ROS 2는 토픽, 서비스, 액션, 파라미터의 네 가지 통신 방식을 제공한다.

### 서비스 vs 액션

| 구분 | 서비스 | 액션 |
|------|--------|------|
| 목적 | 짧은 요청-응답 처리 | 긴 시간 동안 실행되는 작업 처리 |
| 중간 피드백 | 없음 | 있음 (진행 상태 지속 전달) |
| 취소 기능 | 없음 | 있음 |

### 액션 통신 흐름

```
클라이언트 → 요청(Goal) → 서버
서버 → 수락/거부 응답 → 클라이언트
서버 → 피드백(Feedback, 반복) → 클라이언트
서버 → 최종 결과(Result) → 클라이언트
```

예: 로봇이 A → B 지점까지 이동하는 작업(수 초~수 분 소요)에 액션이 적합하다.

## 2. 액션 서버 코드 분석

예제 코드는 ROS 2 Python 예제의 **checker** 노드를 활용한다.  
checker는 기존의 서비스·토픽을 통합하고 **액션 서버**를 포함하는 노드이다.

### 액션 서버 주요 구성 요소

| 요소 | 설명 |
|------|------|
| 액션 서버 클래스 | `rclpy.action.ActionServer` 사용 |
| 액션 타입 | `RSMediChecker` (메시지 타입 정의) |
| 액션 이름 | `rsml_checker` |
| 콜백 함수 | `execute_checker` — 클라이언트 요청 수신 시 실행 |
| 콜백 그룹 | 멀티스레드 병렬 콜백 지원을 위해 설정 |

### `execute_checker` 함수 동작

```
1. total_value(최종 한계값) 및 피드백 메시지(RSMediChecker.Feedback) 초기화
2. while total_sum < goal_sum:
   a. 랜덤 숫자를 total_sum에 누적
   b. 현재 formula를 피드백 메시지로 클라이언트에 전송
   c. 1초 sleep
3. 루프 종료 → goal_handle.succeed() 호출
4. RSMediChecker.Result에 formula 리스트 + total_sum 담아 반환
```

- **`GoalHandle`**: 액션 상태(실행 중, 완료, 취소 등)를 관리한다.
- **피드백 메시지**: 현재까지의 계산 결과(`formula`) 리스트를 포함한다.

### 메인 함수
```python
executor = MultiThreadedExecutor(num_threads=4)
executor.add_node(calculator)
executor.spin()
# 종료 시
executor.shutdown()
rclpy.shutdown()
```

## 3. 액션 클라이언트 코드 분석

액션 클라이언트 코드는 **checker** 폴더 내에 위치한다.

### 액션 클라이언트 주요 기능

| 기능 | 설명 |
|------|------|
| 클라이언트 생성 | `rclpy.action.ActionClient` 사용 |
| 액션 타입·이름 | 서버와 동일: `RSMediChecker`, `rsml_checker` |
| 목표값 설정 | `send_goal_total_sum` 함수로 `goal_sum` 설정 |
| 골 전송 | `send_goal_async` 함수 사용 (비동기) |
| 피드백 수신 | `feedback_callback` 콜백 함수 |
| 결과 수신 | `action_result_callback` 콜백 함수 |

### 서버 접속 대기
```python
while not self.action_client.wait_for_server(timeout_sec=0.1):
    count += 1
    if count >= 3:
        self.get_logger().warn('Action server is not available')
        return
```

### 동기 vs 비동기

| 방식 | 설명 |
|------|------|
| 동기(sync) | 요청 후 응답이 올 때까지 함수가 블로킹됨 |
| 비동기(async) | 요청 후 다른 작업 수행, 응답이 오면 콜백 함수로 처리 |

- 액션 통신은 **비동기 방식**으로 이루어지며 `send_goal_async`를 사용한다.

### 콜백 함수 구성

#### 피드백 콜백 (`feedback_callback`)
- 서버로부터 받은 피드백 메시지를 로깅한다.
- 피드백 메시지에는 서버에서 계산된 `formula` 리스트가 포함된다.

#### 골 콜백 (`goal_callback`)
- 액션 요청에 대한 응답(수락/거부)을 처리한다.
- `goal_handle.is_accepted` 확인 → 수락 시 `action_result_callback` 등록, 거부 시 경고 메시지 출력.

#### 결과 콜백 (`action_result_callback`)
- 액션 완료 후 최종 결과를 처리한다.
- 액션 상태(`status`) 확인 → 성공 시 `formula` 리스트, `total_sum`, 완료 이유 코드(`code`)를 출력한다.

## 4. 클라이언트 메인 함수 및 실행

클라이언트의 메인 함수는 `checker/main.py`에 위치한다.

### argparse를 이용한 실행 인자 처리
```python
import argparse

parser = argparse.ArgumentParser()
parser.add_argument('-g', '--goal_sum', type=int, default=50, help='최종 목표 합계 값')
args = parser.parse_args()
```

- `-g` 또는 `--goal_sum` 인자를 통해 실행 시 최종 목표값(`total_sum`)을 설정할 수 있다.
- 클라이언트 노드는 서버와 달리 **단일 스레드**로 실행된다.

### 클라이언트 실행 흐름
```
argparse로 goal_sum 파싱
  └─ rclpy 초기화
      └─ CheckerActionClient 노드 생성
          └─ send_goal_total_sum(goal_sum) 호출
              └─ send_goal_async → goal_callback
                  └─ 수락 → action_result_callback 등록
                      └─ 피드백 수신 (feedback_callback 반복)
                          └─ 액션 완료 → 결과 출력
  └─ KeyboardInterrupt → rclpy.shutdown()
```
