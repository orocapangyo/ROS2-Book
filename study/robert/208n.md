## 슬라이드 2. 액션 프로그래밍의 개념

먼저 액션의 기본 개념부터 살펴보겠습니다.

액션은 긴 시간 동안 실행되는 작업을 위한 통신 방식입니다.  
서비스처럼 요청과 응답 구조를 가지고 있지만, 단순히 한 번 요청하고 한 번 응답받는 구조가 아닙니다.

액션에서는 클라이언트가 Goal, 즉 목표를 서버에 보냅니다.  
서버는 이 목표를 수락할지 거부할지 먼저 응답합니다.

목표가 수락되면 서버는 작업을 수행하면서 중간 진행 상황을 Feedback으로 계속 전달합니다.  
그리고 작업이 끝나면 최종 결과를 Result로 반환합니다.

즉, 액션은 목표 요청, 중간 피드백, 최종 결과라는 세 가지 흐름을 모두 포함하는 통신 방식입니다.

## 슬라이드 3. 서비스와 액션의 차이

서비스와 액션은 둘 다 클라이언트가 서버에 요청을 보낸다는 점에서는 비슷합니다.

하지만 서비스는 짧은 요청-응답 처리에 적합합니다.  
예를 들어 두 숫자를 더해달라는 요청처럼 결과가 바로 나오는 작업은 서비스로 처리하는 것이 좋습니다.

반면 액션은 시간이 오래 걸리는 작업에 적합합니다.  
작업이 진행되는 동안 중간 상태를 계속 확인할 수 있고, 필요하면 취소도 가능합니다.

서비스에는 중간 피드백이 없습니다.  
요청을 보내면 응답이 올 때까지 기다렸다가 결과만 받습니다.

하지만 액션은 작업 중간에 피드백을 받을 수 있습니다.  
또한 최종적으로 성공했는지, 실패했는지, 어떤 결과가 나왔는지 확인할 수 있습니다.

따라서 단순하고 짧은 작업은 서비스, 오래 걸리고 진행 상황이 필요한 작업은 액션을 사용하는 것이 적절합니다.

## 슬라이드 4. 액션 통신 흐름

액션 통신의 전체 흐름을 단계별로 보겠습니다.

첫 번째로 클라이언트가 서버에게 Goal을 보냅니다.  
Goal은 서버가 수행해야 할 목표값이나 작업 내용을 의미합니다.

두 번째로 서버는 이 Goal을 수락할지 거부할지 판단합니다.  
수락하면 작업을 시작하고, 거부하면 클라이언트에게 거부 응답을 보냅니다.

세 번째로 서버는 작업을 수행하는 동안 Feedback을 반복적으로 보냅니다.  
이 피드백을 통해 클라이언트는 현재 작업이 어디까지 진행되었는지 알 수 있습니다.

네 번째로 작업이 완료되면 서버는 Result를 반환합니다.  
Result에는 최종 결과값, 상태 코드, 완료 정보 등이 포함될 수 있습니다.

즉, 액션은 Goal, Feedback, Result를 중심으로 동작하는 비동기 통신 구조입니다.

## 슬라이드 5. 액션이 필요한 대표 사례

액션이 필요한 대표적인 사례는 로봇 이동입니다.

예를 들어 로봇에게 “A 지점에서 B 지점까지 이동하라”고 명령했다고 가정해보겠습니다.

이 작업은 단순 계산처럼 즉시 끝나지 않습니다.  
몇 초에서 몇 분까지 걸릴 수 있고, 이동 중 장애물을 만나거나 경로가 변경될 수도 있습니다.

이때 클라이언트는 로봇이 이동 중인지, 얼마나 이동했는지, 목표 지점에 도착했는지 알고 싶어 합니다.

서비스만 사용하면 중간 상태를 알 수 없습니다.  
하지만 액션을 사용하면 이동 중 피드백을 받을 수 있고, 최종 도착 결과도 받을 수 있습니다.

그래서 로봇 이동, 경로 계획, 긴 계산 작업, 반복 수행 작업처럼 시간이 걸리는 작업에는 액션이 적합합니다.

## 슬라이드 6. 액션 서버 코드 개요

이번 예제에서는 ROS 2 Python 예제의 `checker` 노드를 중심으로 액션 서버를 살펴봅니다.

`checker` 노드는 기존의 토픽과 서비스 구조를 통합하면서 액션 서버 기능까지 포함하는 노드입니다.

액션 서버는 `rclpy.action.ActionServer` 클래스를 사용해서 생성합니다.

이번 예제에서 사용하는 액션 타입은 `RSMediChecker`입니다.  
액션 이름은 `rsml_checker`로 설정되어 있습니다.

클라이언트가 이 이름으로 Goal을 보내면 서버의 콜백 함수가 실행됩니다.

이때 실행되는 콜백 함수가 `execute_checker`입니다.  
이 함수 안에서 목표값까지 계산을 반복하고, 중간 피드백을 보내고, 최종 결과를 반환합니다.

## 슬라이드 7. 액션 서버 주요 구성 요소

액션 서버를 구성할 때는 몇 가지 요소가 필요합니다.

첫 번째는 액션 서버 클래스입니다.  
ROS 2 Python에서는 `rclpy.action.ActionServer`를 사용합니다.

두 번째는 액션 타입입니다.  
이번 예제에서는 `RSMediChecker`라는 액션 메시지 타입을 사용합니다.

세 번째는 액션 이름입니다.  
서버와 클라이언트가 같은 액션 이름을 사용해야 서로 연결될 수 있습니다.  
이번 예제에서는 `rsml_checker`라는 이름을 사용합니다.

네 번째는 콜백 함수입니다.  
클라이언트 요청이 들어오면 실행되는 함수가 필요합니다.  
여기서는 `execute_checker` 함수가 그 역할을 합니다.

마지막으로 콜백 그룹입니다.  
여러 콜백을 병렬로 처리하기 위해 멀티스레드 콜백 그룹을 설정할 수 있습니다.

## 슬라이드 8. execute_checker 함수 흐름

이제 액션 서버의 핵심인 `execute_checker` 함수의 동작을 살펴보겠습니다.

먼저 클라이언트가 보낸 Goal 값을 확인합니다.  
예제에서는 최종 목표 합계값인 `goal_sum`을 기준으로 작업을 수행합니다.

그다음 현재 누적값인 `total_sum`을 초기화합니다.  
그리고 피드백 메시지 객체를 생성합니다.

이후 반복문이 실행됩니다.  
`total_sum`이 목표값보다 작으면 랜덤 숫자를 계속 더합니다.

값이 더해질 때마다 현재까지의 계산 과정을 formula 형태로 저장합니다.  
그리고 이 formula 리스트를 피드백 메시지로 클라이언트에게 전송합니다.

중간중간 1초씩 대기하면서 작업이 진행되는 것처럼 동작합니다.

마지막으로 누적값이 목표값 이상이 되면 액션을 성공 상태로 표시하고 최종 결과를 반환합니다.

## 슬라이드 9. GoalHandle의 역할

액션 서버에서 중요한 개념 중 하나가 `GoalHandle`입니다.

GoalHandle은 클라이언트가 보낸 목표의 상태를 관리하는 객체입니다.

액션 작업이 실행 중인지, 성공했는지, 취소되었는지 같은 상태 정보를 관리합니다.

예제에서는 작업이 정상적으로 끝났을 때 `goal_handle.succeed()`를 호출합니다.

이 호출은 서버가 “이 Goal은 성공적으로 완료되었다”고 ROS 2 액션 시스템에 알려주는 역할을 합니다.

또한 GoalHandle을 통해 피드백을 발행할 수도 있습니다.  
서버는 작업 중간마다 GoalHandle을 사용해서 클라이언트에게 현재 상태를 전달합니다.

즉, GoalHandle은 액션 서버와 클라이언트 사이에서 작업 상태를 관리하는 핵심 객체입니다.

## 슬라이드 10. 피드백 메시지와 결과 메시지

액션에서 피드백과 결과는 구분해서 이해해야 합니다.

피드백은 작업이 진행되는 동안 반복적으로 전달되는 중간 정보입니다.  
이번 예제에서는 현재까지의 계산 과정인 formula 리스트가 피드백에 포함됩니다.

예를 들어 랜덤 값이 하나씩 누적될 때마다 현재 계산식이 클라이언트에게 전달됩니다.

반면 결과 메시지는 작업이 완전히 끝난 뒤 한 번 반환됩니다.

결과에는 최종 formula 리스트와 최종 누적값인 total_sum이 포함됩니다.

즉, 피드백은 진행 상황을 보여주는 정보이고, 결과는 작업 완료 후 최종 산출물입니다.

이 두 가지를 구분하는 것이 액션 프로그래밍을 이해하는 데 매우 중요합니다.

## 슬라이드 11. 액션 서버 메인 함수

액션 서버의 메인 함수에서는 Executor를 설정합니다.

이번 예제에서는 `MultiThreadedExecutor`를 사용하고, 스레드 수는 4개로 설정합니다.

멀티스레드 Executor를 사용하는 이유는 checker 노드가 액션 서버뿐 아니라 여러 작업을 함께 처리할 수 있기 때문입니다.

예를 들어 토픽을 구독하거나, 서비스를 처리하거나, 액션 요청을 처리하는 일이 동시에 발생할 수 있습니다.

이런 경우 단일 스레드 Executor를 사용하면 하나의 콜백이 끝날 때까지 다른 콜백이 기다려야 합니다.

반면 MultiThreadedExecutor를 사용하면 여러 콜백을 병렬로 처리할 수 있습니다.

마지막으로 프로그램이 종료될 때는 executor를 shutdown하고, rclpy도 shutdown해서 ROS 2 자원을 정리합니다.

## 슬라이드 12. 액션 클라이언트 코드 개요

이제 액션 클라이언트 코드를 살펴보겠습니다.

액션 클라이언트는 checker 폴더 안에 위치합니다.

클라이언트는 `rclpy.action.ActionClient` 클래스를 사용해서 생성합니다.

중요한 점은 서버와 클라이언트가 같은 액션 타입과 같은 액션 이름을 사용해야 한다는 것입니다.

이번 예제에서는 서버와 동일하게 액션 타입은 `RSMediChecker`, 액션 이름은 `rsml_checker`를 사용합니다.

클라이언트는 목표값을 설정한 뒤 `send_goal_async` 함수를 사용해 서버에게 Goal을 보냅니다.

그리고 서버로부터 Goal 수락 여부, 피드백, 최종 결과를 각각 콜백 함수로 처리합니다.

## 슬라이드 13. 액션 클라이언트 주요 기능

액션 클라이언트에는 몇 가지 주요 기능이 있습니다.

첫째, 액션 클라이언트를 생성합니다.  
이때 `ActionClient`를 사용하고, 액션 타입과 액션 이름을 지정합니다.

둘째, 목표값을 설정합니다.  
예제에서는 `send_goal_total_sum` 함수에서 최종 목표 합계값인 `goal_sum`을 설정합니다.

셋째, Goal을 서버로 전송합니다.  
이때 `send_goal_async`를 사용하기 때문에 비동기 방식으로 요청이 전송됩니다.

넷째, 피드백을 수신합니다.  
서버가 작업 중간에 보내는 피드백은 `feedback_callback`에서 처리합니다.

다섯째, 최종 결과를 수신합니다.  
작업 완료 후 결과는 `action_result_callback`에서 처리합니다.

이처럼 액션 클라이언트는 단순 요청자가 아니라, 목표 전송부터 피드백과 결과 처리까지 담당합니다.

## 슬라이드 14. 액션 서버 접속 대기

액션 클라이언트가 Goal을 보내기 전에 먼저 확인해야 할 것이 있습니다.  
바로 액션 서버가 실행 중인지 여부입니다.

이를 위해 `wait_for_server` 함수를 사용합니다.

예제에서는 0.1초 간격으로 서버가 준비되었는지 확인합니다.

서버가 바로 준비되지 않을 수 있기 때문에 일정 횟수 동안 대기합니다.  
만약 3번 이상 확인했는데도 서버를 찾지 못하면 경고 메시지를 출력하고 요청을 중단합니다.

이 과정은 매우 중요합니다.

서버가 없는 상태에서 Goal을 보내면 요청이 실패할 수 있습니다.  
따라서 액션 클라이언트에서는 항상 서버가 준비되어 있는지 확인한 뒤 목표를 전송해야 합니다.

## 슬라이드 15. 동기 방식과 비동기 방식

액션 통신은 기본적으로 비동기 방식으로 처리됩니다.

동기 방식은 요청을 보내고 응답이 올 때까지 함수 실행이 멈추는 방식입니다.  
구조는 단순하지만, 시간이 오래 걸리는 작업에서는 프로그램이 멈춘 것처럼 보일 수 있습니다.

반면 비동기 방식은 요청을 보낸 뒤에도 다른 작업을 계속 수행할 수 있습니다.

액션은 작업 시간이 길어질 수 있기 때문에 비동기 방식이 적합합니다.

이번 예제에서도 `send_goal_async`를 사용합니다.  
Goal을 보낸 뒤 서버가 수락했는지, 피드백이 왔는지, 결과가 도착했는지를 콜백 함수로 처리합니다.

즉, 액션 프로그래밍에서는 비동기 흐름과 콜백 구조를 이해하는 것이 핵심입니다.

---

## 슬라이드 16. 피드백 콜백 함수

피드백 콜백 함수는 서버가 보내는 중간 진행 상황을 처리합니다.

이번 예제에서는 `feedback_callback` 함수가 그 역할을 합니다.

서버는 작업 중간마다 현재까지의 계산 과정을 피드백으로 보냅니다.

클라이언트는 이 피드백을 받아 로그로 출력합니다.

피드백 메시지에는 formula 리스트가 포함됩니다.  
이 리스트를 보면 현재까지 어떤 값들이 누적되었고, 계산이 어떻게 진행되고 있는지 확인할 수 있습니다.

피드백 콜백은 액션의 장점을 가장 잘 보여주는 부분입니다.

서비스에서는 중간 상태를 알 수 없지만, 액션에서는 피드백 콜백을 통해 작업 진행 상황을 계속 확인할 수 있습니다.

---

## 슬라이드 17. Goal 콜백 함수

Goal 콜백 함수는 클라이언트가 보낸 목표 요청에 대해 서버가 수락했는지 거부했는지를 처리합니다.

예제에서는 `goal_callback` 함수가 이 역할을 합니다.

먼저 서버의 응답에서 GoalHandle을 가져옵니다.  
그리고 `goal_handle.is_accepted` 값을 확인합니다.

만약 Goal이 수락되었다면, 클라이언트는 최종 결과를 받기 위한 콜백을 등록합니다.

반대로 Goal이 거부되었다면 경고 메시지를 출력하고 더 이상 작업을 진행하지 않습니다.

이 단계는 액션 통신에서 매우 중요합니다.  
왜냐하면 Goal이 수락되어야만 피드백과 결과를 받을 수 있기 때문입니다.

즉, Goal 콜백은 액션 작업이 실제로 시작될 수 있는지를 판단하는 관문 역할을 합니다.

---

## 슬라이드 18. 결과 콜백 함수

결과 콜백 함수는 액션 작업이 완료된 뒤 최종 결과를 처리합니다.

이번 예제에서는 `action_result_callback` 함수가 이 역할을 합니다.

먼저 액션 상태 값을 확인합니다.  
상태 값은 작업이 성공했는지, 실패했는지, 취소되었는지를 알려줍니다.

작업이 성공적으로 완료되었다면 결과 메시지를 읽습니다.

결과 메시지에는 최종 formula 리스트와 total_sum 값이 포함됩니다.  
또한 완료 이유를 나타내는 code 값도 출력할 수 있습니다.

이 결과 콜백은 액션 통신의 마지막 단계입니다.

즉, 클라이언트가 Goal을 보내고, 피드백을 계속 받은 뒤, 최종적으로 작업 결과를 확인하는 지점입니다.

---

## 슬라이드 19. 클라이언트 메인 함수 구조

액션 클라이언트의 메인 함수는 `checker/main.py`에 위치합니다.

이 메인 함수에서는 먼저 실행 인자를 처리합니다.

예제에서는 `argparse`를 사용해 `goal_sum` 값을 입력받습니다.

사용자는 실행할 때 `-g` 또는 `--goal_sum` 옵션을 사용해서 최종 목표 합계값을 지정할 수 있습니다.

예를 들어 목표값을 100으로 설정하면 서버는 total_sum이 100 이상이 될 때까지 작업을 수행합니다.

그다음 rclpy를 초기화하고, CheckerActionClient 노드를 생성합니다.

그리고 `send_goal_total_sum` 함수를 호출해서 액션 서버에 목표값을 전송합니다.

## 슬라이드 20. argparse를 이용한 실행 인자 처리

이번 예제에서는 Python의 `argparse` 모듈을 사용합니다.

`argparse`는 프로그램 실행 시 사용자가 입력한 옵션값을 처리하는 표준 라이브러리입니다.

예제에서는 `-g` 또는 `--goal_sum` 인자를 정의합니다.

이 인자는 최종 목표 합계값을 의미합니다.  
기본값은 50으로 설정되어 있습니다.

즉, 사용자가 별도 값을 입력하지 않으면 goal_sum은 50으로 동작합니다.

하지만 실행 시 `--goal_sum 100`처럼 입력하면 목표값을 100으로 변경할 수 있습니다.

이렇게 실행 인자를 사용하면 코드를 수정하지 않고도 테스트 조건을 쉽게 바꿀 수 있습니다.

## 슬라이드 21. 액션 클라이언트 실행 흐름

액션 클라이언트 실행 흐름을 전체적으로 정리해보겠습니다.

먼저 argparse로 goal_sum 값을 파싱합니다.

그다음 `rclpy.init()`으로 ROS 2 Python 환경을 초기화합니다.

이후 `CheckerActionClient` 노드를 생성합니다.

그리고 `send_goal_total_sum(goal_sum)`을 호출해서 액션 서버에 Goal을 보냅니다.

서버가 Goal을 수락하면 goal_callback에서 결과 콜백을 등록합니다.

작업이 진행되는 동안 feedback_callback이 반복적으로 호출됩니다.

마지막으로 액션이 완료되면 action_result_callback에서 최종 결과를 출력합니다.

사용자가 Ctrl+C를 누르면 KeyboardInterrupt가 발생하고, `rclpy.shutdown()`을 호출해 정상 종료합니다.

## 슬라이드 22. 액션 프로그래밍 전체 정리

이번 내용을 정리해보겠습니다.

액션은 시간이 오래 걸리는 작업을 처리하기 위한 ROS 2 통신 방식입니다.

액션은 Goal, Feedback, Result 구조로 동작합니다.

클라이언트는 Goal을 보내고, 서버는 이 Goal을 수락하거나 거부합니다.

작업이 진행되는 동안 서버는 Feedback을 반복적으로 전달합니다.

작업이 끝나면 서버는 Result를 반환합니다.

이번 예제에서는 checker 노드를 통해 목표 합계값까지 랜덤 값을 누적하는 액션 구조를 살펴보았습니다.

서버는 `ActionServer`를 사용하고, 클라이언트는 `ActionClient`를 사용합니다.

또한 액션은 비동기 방식으로 동작하므로 콜백 함수의 흐름을 잘 이해해야 합니다.
