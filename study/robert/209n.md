## 슬라이드 2. 파라미터의 개념과 활용

먼저 파라미터의 개념부터 보겠습니다.

파라미터는 노드가 실행될 때 사용하는 설정값입니다.  
예를 들어 `min_random_number`와 `max_random_number`를 파라미터로 만들면, 랜덤 숫자의 생성 범위를 실행 중에도 조정할 수 있습니다.

ROS 2에서 파라미터를 다루는 방법은 여러 가지가 있습니다.

세 번째는 서비스 클라이언트를 이용해 외부 노드에서 파라미터를 set 또는 get 하는 방식입니다.  
네 번째는 YAML 파일을 사용해 노드 실행 시 파라미터를 일괄 초기화하는 방식입니다.

즉, 파라미터는 ROS 2 시스템을 더 유연하고 재사용 가능하게 만들어주는 중요한 기능입니다.

## 슬라이드 3. 서비스와 파라미터의 차이

파라미터도 내부적으로는 서비스 기반으로 동작합니다.  
하지만 일반 서비스와는 몇 가지 차이가 있습니다.

일반 서비스는 사용자가 직접 `.srv` 파일을 정의해서 요청과 응답 구조를 만듭니다.  
반면 파라미터는 ROS 2에서 이미 정의된 파라미터 인터페이스를 사용합니다.

즉, 사용자가 별도의 메시지 인터페이스를 만들지 않아도 파라미터를 조회하고 변경할 수 있습니다.

서비스는 특정 작업을 요청하고 결과를 받는 구조입니다.  
반면 파라미터는 노드가 가지고 있는 설정값을 읽거나 변경하는 구조입니다.

예를 들어 “계산을 수행해줘”는 서비스에 가깝고, “랜덤 숫자 최대값을 100으로 바꿔줘”는 파라미터 변경에 가깝습니다.

이 차이를 이해하면 어떤 상황에서 서비스와 파라미터를 각각 사용해야 하는지 판단할 수 있습니다.

## 슬라이드 4. Python 코드의 파라미터 구성 요소

ROS 2 Python 코드에서 파라미터를 다룰 때 주로 사용하는 함수는 세 가지입니다.

첫 번째는 `declare_parameter`입니다.  
이 함수는 노드에서 사용할 파라미터를 미리 선언합니다.

두 번째는 `get_parameter`입니다.  
이 함수는 선언된 파라미터의 현재 값을 읽어옵니다.  
만약 외부에서 설정된 값이 없다면 기본값을 사용합니다.

세 번째는 `add_on_set_parameter_callback`입니다.  
이 함수는 외부에서 파라미터가 변경될 때 실행할 콜백 함수를 등록합니다.

이 세 함수는 모두 노드 클래스 안에서 사용하는 멤버 함수입니다.

정리하면, 파라미터는 먼저 선언하고, 값을 읽고, 변경될 때 콜백으로 대응하는 흐름으로 사용됩니다.

## 슬라이드 5. argument 노드의 파라미터 선언

이번 예제에서는 `argument` 노드에서 파라미터를 사용하는 구조를 살펴봅니다.

먼저 `declare_parameter`를 사용해서 `qos_depths`라는 파라미터를 선언합니다.  
기본값은 10으로 설정합니다.

그다음 `get_parameter`를 통해 현재 파라미터 값을 읽어옵니다.  
YAML 파일이나 CLI에서 설정된 값이 있다면 그 값이 사용되고, 별도 설정이 없다면 기본값 10이 사용됩니다.

마지막으로 `add_on_set_parameter_callback`을 사용해 파라미터 변경 콜백을 등록합니다.

이렇게 하면 노드 실행 중 외부에서 파라미터 값이 바뀌었을 때, 노드 내부에서 그 변경을 감지하고 처리할 수 있습니다.

즉, 파라미터 선언은 단순한 변수 선언이 아니라, 노드 외부와 설정값을 연결하는 과정입니다.

## 슬라이드 6. 주요 파라미터: qos_depths, min, max

이번 예제의 `argument` 노드에서는 대표적으로 세 가지 파라미터를 사용합니다.

첫 번째는 `qos_depths`입니다.  
이 값은 토픽 메시지의 QoS depth를 설정하는 데 사용됩니다.  
즉, 메시지를 몇 개까지 보관할 것인지와 관련된 설정입니다.

두 번째는 `min_random_number`입니다.  
랜덤 숫자를 생성할 때 최소값으로 사용됩니다.

세 번째는 `max_random_number`입니다.  
랜덤 숫자를 생성할 때 최대값으로 사용됩니다.

예를 들어 `min_random_number`가 0이고 `max_random_number`가 9라면, 노드는 0부터 9 사이의 랜덤 숫자를 생성합니다.

이 값을 파라미터로 관리하면 코드를 수정하지 않고도 실행 중 숫자 생성 범위를 바꿀 수 있습니다.

## 슬라이드 7. update_parameters 콜백 함수

파라미터가 외부에서 변경되면 등록된 콜백 함수가 실행됩니다.

이번 예제에서는 `update_parameters` 함수가 그 역할을 합니다.

이 함수는 변경 요청이 들어온 파라미터 리스트를 전달받습니다.  
각 파라미터에는 이름과 타입, 값 정보가 포함되어 있습니다.

함수 안에서는 파라미터 이름을 확인합니다.  
`min_random_number`가 변경되었다면 노드의 최소값 멤버 변수를 업데이트합니다.  
`max_random_number`가 변경되었다면 최대값 멤버 변수를 업데이트합니다.

또한 값의 유효성을 검사할 수도 있습니다.  
예를 들어 최대값이 최소값보다 작거나, 허용 범위를 벗어난 값이 들어오면 변경을 거부할 수 있습니다.

마지막으로 `SetParametersResult(successful=True)`를 반환하면 파라미터 변경이 성공적으로 처리됩니다.

## 슬라이드 8. CLI를 이용한 파라미터 관리

ROS 2에서는 CLI 명령어로 실행 중인 노드의 파라미터를 확인하고 변경할 수 있습니다.

먼저 노드를 실행합니다.  
예를 들어 `ros2 run topic_service_examples arguments` 명령으로 argument 노드를 실행할 수 있습니다.

기본적으로 이 노드는 `min_random_number=0`, `max_random_number=9` 값을 사용해서 0부터 9 사이의 랜덤 숫자를 생성합니다.

이 상태에서 다른 터미널을 열고 `ros2 param` 명령어를 사용할 수 있습니다.

`ros2 param list`는 실행 중인 노드의 파라미터 목록을 보여줍니다.  
`ros2 param get`은 특정 파라미터의 타입과 현재 값을 확인합니다.  
`ros2 param set`은 특정 파라미터 값을 변경합니다.

CLI 방식은 테스트와 디버깅할 때 매우 편리합니다.

## 슬라이드 9. ros2 param set 예시

이번에는 실제 파라미터 변경 예시를 보겠습니다.

예를 들어 `max_random_number`를 100으로 바꾸고 싶다면 다음과 같이 입력합니다.

`ros2 param set /arguments max_random_number 100`

명령이 성공하면 `Set parameter successful` 메시지가 출력됩니다.

이후 `argument` 노드는 기존의 0부터 9 범위가 아니라, 0부터 100 사이의 랜덤 값을 생성하게 됩니다.

여기서 중요한 점은 이 변경이 실행 중인 노드의 메모리상에서 적용된다는 것입니다.

즉, 노드를 종료하고 다시 실행하면 YAML 파일이나 기본값으로 다시 초기화될 수 있습니다.

따라서 일시적인 테스트는 CLI로 하고, 지속적으로 사용할 설정값은 YAML 파일에 저장하는 것이 좋습니다.

## 슬라이드 10. 서비스 클라이언트를 이용한 파라미터 관리

ROS 2의 각 노드는 자체적인 파라미터 서버를 가지고 있습니다.

외부 노드는 서비스 클라이언트를 통해 이 파라미터 서버에 접근할 수 있습니다.

예를 들어 `arguments` 노드의 파라미터를 바꾸고 싶다면 `/arguments/set_parameters`라는 서비스에 요청을 보낼 수 있습니다.

Python 코드에서는 `rcl_interfaces.srv`의 `SetParameters`를 사용합니다.

또한 파라미터 값을 표현하기 위해 `Parameter`, `ParameterValue`, `ParameterType` 같은 메시지 타입을 사용합니다.

이 방식은 다른 노드가 실행 중인 노드의 설정을 동적으로 변경해야 할 때 유용합니다.

즉, CLI가 사람이 터미널에서 직접 변경하는 방식이라면, 서비스 클라이언트 방식은 프로그램이 자동으로 파라미터를 변경하는 방식입니다.

## 슬라이드 11. 파라미터 설정 요청 구조

서비스 클라이언트를 이용해 파라미터를 변경하려면 먼저 요청 객체를 만들어야 합니다.

예제에서는 `SetParameters.Request()`를 생성합니다.

그다음 변경할 파라미터 값을 `ParameterValue`로 만듭니다.

예를 들어 정수형 파라미터라면 `ParameterType.PARAMETER_INTEGER` 타입을 지정하고, `integer_value`에 실제 값을 넣습니다.

이후 `Parameter` 객체를 만들고, 이름에는 변경할 파라미터 이름을 넣습니다.  
예를 들어 `max_random_number`를 지정할 수 있습니다.

마지막으로 이 Parameter 객체를 request의 parameters 리스트에 넣습니다.

그리고 `call_async`를 사용해 비동기 방식으로 파라미터 변경 요청을 보냅니다.

## 슬라이드 12. 비동기 파라미터 변경의 장점

파라미터를 서비스 클라이언트로 변경할 때는 주로 비동기 방식을 사용합니다.

비동기 방식은 요청을 보낸 뒤 응답이 올 때까지 프로그램 전체가 멈추지 않습니다.

ROS 2 시스템에서는 여러 노드와 콜백이 동시에 동작하는 경우가 많습니다.  
따라서 파라미터 변경 요청 하나 때문에 전체 노드가 멈추는 것은 좋지 않습니다.

`call_async`를 사용하면 파라미터 변경 요청을 보내고, 다른 작업을 계속 수행할 수 있습니다.

응답이 오면 future 객체나 콜백을 통해 결과를 확인하면 됩니다.

이 방식은 실시간성이나 반응성이 중요한 로봇 시스템에서 특히 유용합니다.

## 슬라이드 13. YAML 파일을 이용한 파라미터 초기화

이번에는 YAML 파일을 이용한 파라미터 초기화 방법을 보겠습니다.

YAML 파일은 노드를 실행할 때 파라미터를 일괄 적용하는 설정 파일입니다.

예를 들어 YAML 파일 안에 `qos_depths`, `min_random_number`, `max_random_number` 값을 미리 작성해둘 수 있습니다.

노드를 실행할 때 이 파일을 읽도록 설정하면, 코드의 기본값 대신 YAML에 정의된 값이 사용됩니다.

자료의 예시에서는 `qos_depths`를 30으로 설정하고, 랜덤 숫자 범위는 0부터 9로 설정합니다.

YAML 파일을 사용하면 여러 환경의 설정을 파일 단위로 관리할 수 있습니다.

개발용, 테스트용, 실제 로봇 운용용 파라미터 파일을 따로 만들어두는 식으로 활용할 수 있습니다.

## 슬라이드 14. YAML 구조와 /** 의미

YAML 파일에서 중요한 구조가 있습니다.

예제에서는 `/**:` 형태가 사용됩니다.

이 표기는 현재 설정된 네임스페이스나 노드 이름에 자동으로 적용될 수 있도록 해주는 방식입니다.

즉, 노드 이름이나 네임스페이스가 바뀌더라도 파라미터가 유연하게 매핑될 수 있습니다.

그 아래에는 `ros__parameters` 항목이 들어갑니다.

그리고 그 하위에 실제 파라미터 이름과 값이 작성됩니다.

이 구조를 잘 지켜야 ROS 2가 YAML 파일을 파라미터 설정 파일로 올바르게 읽을 수 있습니다.

## 슬라이드 15. CLI 변경과 YAML 초기값의 차이

여기서 중요한 점이 하나 있습니다.

CLI로 변경한 파라미터 값은 실행 중인 노드의 메모리상에서 변경됩니다.

즉, 현재 실행 중인 노드에는 바로 반영되지만, 파일에 저장되는 것은 아닙니다.

따라서 노드를 종료했다가 다시 실행하면 CLI로 변경했던 값은 사라지고, YAML 파일이나 코드의 기본값으로 다시 초기화됩니다.

반면 YAML 파일에 작성된 값은 노드 실행 시마다 초기값으로 적용됩니다.

그래서 임시 테스트는 CLI를 사용하고, 지속적으로 유지할 설정은 YAML 파일에 기록하는 것이 좋습니다.

이 차이를 이해해야 파라미터 변경 후 재실행했을 때 값이 왜 되돌아가는지 혼동하지 않을 수 있습니다.

---

## 슬라이드 16. 파라미터 파일 위치

예제에서 사용하는 파라미터 파일은 `examples/checker/parameter/rs_medic_config.yml` 경로에 위치합니다.

일반적으로 ROS 2 패키지에서는 파라미터 파일을 `parameter` 또는 `config` 폴더 안에 관리합니다.

이렇게 폴더를 분리해두면 launch 파일에서 파라미터 파일을 참조하기 쉽고, 패키지 구조도 명확해집니다.

중요한 것은 이 파일이 빌드 후 install 공간에도 복사되어야 한다는 점입니다.

소스 폴더에만 파일이 있으면 개발 중에는 보일 수 있지만, ROS 2 실행 환경에서는 찾지 못할 수 있습니다.

따라서 setup.py의 `data_files` 설정을 통해 파라미터 파일이 설치 경로로 복사되도록 해야 합니다.

---

## 슬라이드 17. 런치 파일을 통한 파라미터 설정

ROS 2에서는 launch 파일을 사용해서 노드를 실행할 때 파라미터 파일을 함께 지정할 수 있습니다.

예제에서는 `LaunchConfiguration`을 사용해 파라미터 파일 경로를 지정합니다.

기본 경로는 `get_package_share_directory`를 통해 패키지의 share 폴더를 찾고, 그 안의 parameter 폴더에 있는 YAML 파일을 참조합니다.

그다음 `Node` 설정 안에서 `parameters=[param_file]` 형태로 YAML 파일을 전달합니다.

이렇게 하면 노드가 실행될 때 해당 YAML 파일을 읽어 파라미터를 초기화합니다.

즉, launch 파일은 여러 노드를 실행하는 것뿐만 아니라 각 노드의 설정 파일까지 함께 연결하는 역할을 합니다.

실제 프로젝트에서는 launch 파일과 YAML 파일을 함께 사용하는 구조가 가장 일반적입니다.

---

## 슬라이드 18. setup.py의 data_files 설정

마지막으로 패키지 빌드와 설치 과정에서 파라미터 파일을 어떻게 포함시키는지 보겠습니다.

ROS 2 Python 패키지에서는 `setup.py`의 `data_files` 항목을 사용합니다.

여기에는 launch 파일과 parameter 파일을 install 공간의 share 폴더로 복사하도록 설정합니다.

예를 들어 launch 파일은 `share/<package_name>/launch` 경로로 복사합니다.

파라미터 파일은 `share/<package_name>/parameter` 경로로 복사합니다.

이를 위해 Python의 `glob` 함수를 사용해서 `launch/*.launch.py`, `parameter/*.yaml` 같은 파일 패턴을 지정할 수 있습니다.

이 설정이 누락되면 빌드는 되더라도 launch 실행 시 YAML 파일을 찾지 못하는 문제가 발생할 수 있습니다.

따라서 파라미터 파일을 사용하는 패키지라면 setup.py의 data_files 설정을 반드시 확인해야 합니다.

## 슬라이드 19. 전체 흐름 정리

이번 파라미터 프로그래밍의 전체 흐름을 정리해보겠습니다.

먼저 Python 코드에서 `declare_parameter`로 파라미터를 선언합니다.

그다음 `get_parameter`로 현재 값을 읽어와 노드 동작에 사용합니다.

외부에서 파라미터가 변경될 수 있다면 `add_on_set_parameter_callback`으로 변경 콜백을 등록합니다.

실행 중에는 `ros2 param` CLI 명령어로 파라미터를 조회하거나 변경할 수 있습니다.

또는 서비스 클라이언트를 이용해서 외부 노드가 파라미터를 동적으로 변경할 수도 있습니다.

초기 설정값은 YAML 파일로 관리하고, launch 파일에서 이 YAML 파일을 노드에 전달합니다.

마지막으로 setup.py의 data_files에 YAML 파일을 포함시켜 빌드 후에도 정상적으로 참조되도록 해야 합니다.
